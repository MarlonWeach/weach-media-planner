# Task 10-8: Fluxo de senha para utilizadores criados só via Google OAuth

## Status
✅ Done

## Decisão
Implementar **flag explícita na base** (`senhaLocalConfigurada`) em vez de heurísticas sobre `googleSub` ou hash: é previsível, auditável e segura no servidor.

- **`senhaLocalConfigurada = true`** (default): comportamento actual — alterar senha exige **senha atual**.
- **`senhaLocalConfigurada = false`**: conta criada pelo **primeiro login Google** (auto-provisionamento); o utilizador autenticado com JWT pode **definir a primeira senha** sem “senha atual”. Depois do `POST /api/auth/change-password`, o campo passa a `true`.

Quem **já tinha senha** e associa Google mantém `senhaLocalConfigurada === true` (não alteramos no link OAuth).

## Implementação (resumo)
- **Prisma:** `wp_Usuario.senhaLocalConfigurada Boolean @default(true)`.
- **OAuth callback:** `create` com `senhaLocalConfigurada: false`.
- **`POST /api/auth/change-password`:** se `senhaLocalConfigurada`, valida `senhaAtual`; caso contrário ignora; no fim grava `senhaLocalConfigurada: true`.
- **`GET /api/auth/me`** e **`POST /api/auth/login`:** expõem `senhaLocalConfigurada` ao cliente.
- **`/conta/ajustes`:** omite campo “Senha atual” quando `senhaLocalConfigurada === false`; após sucesso chama `refreshUsuario()`.
- **Onboarding / admin:** ao definir senha por fluxo normal, `senhaLocalConfigurada: true` (create/update).

## Contas criadas por OAuth **antes** desta migração
A coluna nova assume **default `true`** para linhas existentes. Quem já era “só Google” antes do deploy pode ficar com `true` incorrecto até:

- **Opção A:** definir manualmente no SQL: `UPDATE "wp_Usuario" SET "senhaLocalConfigurada" = false WHERE id = '<uuid>';` (só para contas que **nunca** tiveram senha escolhida pelo utilizador).
- **Opção B:** apagar/recriar o utilizador de teste.

Não usar `UPDATE ... WHERE "googleSub" IS NOT NULL` em massa: utilizadores que **tinham** senha e depois ligaram Google continuam a precisar de “senha atual”.
