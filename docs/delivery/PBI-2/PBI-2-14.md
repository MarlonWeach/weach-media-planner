# Task 2-14: Criar onboarding de senha para solicitante via e-mail com link de definição de senha

## Status
✅ Done

## Descrição
Ao cadastrar um novo solicitante em `/admin/cadastros`, o sistema deve enviar automaticamente um e-mail de convite informando que ele foi cadastrado no sistema de cotações da Weach e que precisa criar sua senha através de um link seguro.

## Objetivo
Permitir ativação autônoma de acesso para solicitantes cadastrados, reduzindo dependência de criação manual de senha por administradores.

## Fluxo Esperado
1. Usuário autorizado cadastra solicitante em `/admin/cadastros`.
2. Sistema envia e-mail para o `email do solicitante` com link de criação de senha.
3. Solicitante acessa o link e vai para página de definição de senha.
4. Na página, informa o e-mail e define a nova senha.
5. Senha é persistida com hash e usuário fica apto a autenticar no sistema.

## Regras de Acesso
- `/admin/cadastros`: acessível por `ADMIN` e `MANAGER`.
- `/admin/usuarios`: acessível somente por `ADMIN`.

## Critérios de Aceite
- [x] Cadastro de solicitante em `/admin/cadastros` dispara e-mail automático para o e-mail informado.
- [x] E-mail contém mensagem de boas-vindas e link funcional para criação de senha.
- [x] Link possui validade e mecanismo seguro (token único/expirável).
- [x] Página de criação de senha solicita e-mail + nova senha + confirmação.
- [x] Página valida token/expiração e impede uso inválido.
- [x] Senha é armazenada em formato hash seguro.
- [x] Solicitante consegue autenticar após concluir a criação de senha.
- [x] Regras de acesso de `/admin/cadastros` e `/admin/usuarios` são validadas e respeitadas.

## Observações
- Reaproveitar infraestrutura SMTP já utilizada no projeto.
- Evitar divulgar informações sensíveis no link (usar token, não senha).
- Mensagens de erro/sucesso devem ser claras para usuário final.

## Implementação
- Criado envio de onboarding por e-mail no cadastro de solicitante:
  - `lib/notifications/onboardingEmail.ts`
  - `app/api/admin/solicitantes/route.ts`
- Token seguro de onboarding persistido em `wp_Configuracao` (chave `onboarding_solicitante:<token>`) com expiração e marcação de uso.
- Criada API de conclusão do onboarding:
  - `POST /api/auth/complete-onboarding`
  - valida token, e-mail, expiração e uso único
  - cria/atualiza usuário com senha em hash.
- Criada página pública de definição de senha:
  - `app/definir-senha/page.tsx`
- Ajustadas permissões de cadastros para `ADMIN` e `MANAGER`:
  - frontend: `app/admin/cadastros/page.tsx`
  - APIs: `app/api/admin/solicitantes/*` e `app/api/admin/agencias/*`
- Mantida regra de `admin/usuarios` exclusivo para `ADMIN`.
